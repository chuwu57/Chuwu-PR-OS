# final-review.md

本文件用于 press-release 的终稿阶段。  
本文件是终稿阶段唯一风险展示规则来源。终稿阶段的基础成稿检查仍需服从 `SKILL.md` 通用底线、`references/writing-rules.md` 写作失败条件和当前主类型对应类型卡的类型失败条件。

1. 终稿检查启动条件

修改达到 2 轮后，主动询问用户：
是否需要进入终稿检查？
用户只要求"发布前检查""帮我终审""风险检查""终审"时，启动终稿风险检查。
用户要求"终稿""正式版""正式全文""可直接发布版本""可发布版本"时，应先按 `references/writing-rules.md` 完成或整理正文，再启动终稿风险检查。

2. 终稿检查顺序

进入终稿检查后，必须按以下顺序执行：
先确认稿件已完成基础成稿检查：
检查 `SKILL.md` 中的"通用底线"。
检查当前主类型对应类型卡中的"类型失败条件"。
检查 `references/writing-rules.md` 中对的通用失败条件。
如基础检查发现问题，先完成修改，不得直接进入终稿风险输出。基础检查完成后，执行本文件中的终稿风险检查。

3. 类型失败条件调取说明

终稿检查时，必须根据当前稿件主类型，回到对应类型卡检查"类型失败条件"。
新品发布类：检查 `references/types/new-product.md` 中的"类型失败条件"。
宠物公益类：检查 `references/types/pet-welfare.md` 中的"类型失败条件"。
活动举办类：检查 `references/types/event-news.md` 中的"类型失败条件"。
专家观点类：检查 `references/types/expert-view.md` 中的"类型失败条件"。
全年回顾类：检查 `references/types/year-review.md` 中的"类型失败条件"。

4. 审稿防火墙

写作过程中必须内部检查。日常输出不展示全部检查过程。只有终稿阶段，才展示黄色和红色风险项；全绿项不展示。
交稿前失败条件优先级高于审稿防火墙。命中失败条件时，先改稿；未命中失败条件但存在表达风险时，再按审稿防火墙提示黄色或红色风险。
风险项：
传播对象不明：动笔前必须明确 to C / to B / to G / 混合，标题和导语必须体现对象感知。
群体对立风险：禁用制造养宠人、非养宠人、品牌方、经销体系、平台或行业之间对立的措辞，不拉踩、不站队、不标签化。
平台逐利感过强：商业数据后置或弱化，优先写养宠人、宠物福利、经销体系、行业或社会获益。
自夸自嗨：禁用"行业首创""领先""率先""标杆"等自封词，用第三方数据、权威来源或可验证事实替代。
黑话和理解门槛：内部术语必须翻译成公众语言。
政策口径不规范：涉政策引述必须逐字核对原文，不得意译、扩写或借政策自夸。
社会治理价值不足：to G 稿主体段必须有治理价值、规则机制或公共价值呼应。
养宠人获得感不足：to C 稿必须写清养宠人得到了什么，在哪个场景能感知。
经销体系价值不清：to B 稿必须有经销体系获益的具体案例、数据、工具、规则或机制。
高调冒进：表态克制，不承诺未落地计划。

5. 风险等级

绿色：无明显问题，不展示。
黄色：存在轻微风险或信息缺口，可通过改写、补口径、补案例解决。
红色：存在重大事实、口径、政策或结构风险，必须修改后再交付终稿。

6. 终稿输出规则

用户确认进入终稿阶段后，输出必须包含：
终稿正文。
黄色风险项，如有。
红色风险项，如有。
如无黄色和红色风险，只写"未发现需提示的黄色或红色风险项"。不得展示全部绿色检查项。

# writing-rules.md

本文件用于 press-release 的写作阶段，包括 MVP、完整通稿、局部内容、改稿和结构建议。  
本文件只处理写作执行规则。终稿风险展示由 `final-review.md` 处理。五类稿件（新品发布、宠物公益、活动举办、专家观点、全年回顾）的类型路径和类型失败条件由 `references/types/*.md` 处理。

1. 写作前判断

写作前必须完成以下内部判断，后台执行全部素材梳理任务，但不得在消息框内展示任何梳理过程，除非用户要求说明思路：
核心新闻点：一句话写清"谁做了什么，对谁有什么变化"。
标题主打：数据 / 新品 / 活动结果 / 公益动作 / 专家判断 / 年度发现，只选一个。
导语钩子：第一句应前置的核心事实、数据、动作或场景。
正文路径：按当前主类型选择，不得临时换结构。
类型卡确认：已读取当前类型卡全文（含正文路径和类型失败条件），路径已标注。未读类型卡不得动笔。
本稿禁写项：识别最容易跑偏的 1—3 个问题。
短稿 / 长稿判断：根据素材密度判断输出 800—1000 字短稿，或 1000—1500 字长稿。
继承关系判断：如已有用户确认的 MVP 方向，完整稿必须继承。
短稿：800—1000 字。不强行设置小标题。适用于案例单一、数据维度不足、新闻点较窄的素材。
长稿：1000—1500 字。必须设置 2 个小标题，每个小标题下正文不得少于 500 字。适用于素材充足、至少能展开两个维度的素材。
必要时再问：有没有必须保留的标题方向、数据、口径、人物称谓或不可改动内容。

MVP 门控（动笔前硬规则）：
首次输出必须走 MVP，无例外。
新增素材后，核心新闻点改变的→重新出 MVP。
完整稿必须继承已确认 MVP 的方向。

完整稿关键事实核实（动笔前硬规则）：
完整稿写作前，时间、数据、机制细节等关键事实必须重新从素材/知识库核实，不得依赖MVP阶段的记忆。缺少的时间/数据标注【待补】，全文不得超过一处。

倒查检查点（动笔前+交付前各一次）：
动笔前：检查标题、小标题是否含"对象+动作+数据/机制/结果"至少三项、是否有新闻性的完整陈述句、每节是否超过500字。不通过不得动笔。每节不足500字且无素材可补时，降级为短稿（不设小标题）。
交付前：逐条核对第3节（共享规则）+ 第7节（通用失败条件）+ 类型卡类型失败条件。命中项自行修改，倒查过程及结果均不输出，仅交付最终稿件。

2. 渠道校准

用户提供参考稿时，优先按参考稿校准标题句式、导语节奏、段落密度、叙述视角和媒体语感；不得照搬事实、数据、案例、引语或结论。
未指定具体媒体时，不搜索，默认主流综合媒体硬新闻风格：事实前置、语气克制、标题稳健、少评论、少营销腔、多动作、多数据、多机制、多可引用信息。
指定渠道类型时，按类型调整：
党央媒 / 综合媒体：语气平稳，数据和机制前置，强调公共价值和事实进展。
都市报 / 区域媒体：突出本地、人物、场景、养宠家庭、门店，数据尽量前置，可适度增强现场感。
行业媒体：突出行业规则、供给变化、经营效率、产业链协同和机制细节，行业术语必须翻译成外部读者能理解的语言。
指定具体媒体或要求贴近某媒体风格时，优先使用用户提供的参考稿。若运行环境支持搜索，搜索目标媒体近期同类报道，提取 5 条以上真实标题，归纳句式、长度、主体、动词和数据使用方式后再写。若不支持搜索，不得凭印象模仿，只按渠道类型校准。

3. 共享写作规则

3.1 标题

标题之间必须方向不同，不得只是同义改写。标题不得只有活动名、价值判断、宣传口号或抽象概念。标题应尽量体现传播对象，让读者能判断这条稿件主要面向谁。

3.2 导语

导语不得只是复述标题，应补齐基本 5W。导语后接无信息量引语时删除，只有引语能说明下一步、机制、范围扩展时才保留。
重要数据后必须有一句话说明变化，不得连续罗列 3 个以上数据不解释。素材不足以支撑完整稿时，不得硬写长稿，应说明缺口或降级为短稿方向。

3.3 正文

完整通稿总字数不超过 3000 字，超过必须删减。小标题必须对应独立新闻点，须是本次稿件新增信息，如新政策、新数据、新事件。小标题必须是新闻陈述句，不能是价值判断、散文句、悬念句、破折号句，突出核心数据；用单个案例命名小标题时，改为反映整体维度。
小标题和主标题一样要有新闻性：含"对象+动作+数据/机制/结果"至少三项。只有现象描述、只有趋势判断、只有品类名称的小标题视为失败，必须重写或删除该节。
正文按新闻逻辑组织：新闻点 → 数据 / 措施 / 案例 → 机制 / 引语 / 展望，不得按素材顺序机械拼接。正文篇幅按新闻主线分配，核心新闻点和主体维度占主要篇幅，案例、机制、背景、引语用于支撑主线，不得反客为主。无法支撑两个以上独立维度时，不强行设置小标题。背景、沿革、长期机制不得单独成节，应融入中后段。

小节字数硬约束（类型失败条件）：每个小标题下的正文字数（不含标题本身）不得少于 500 字。不足 500 字的节，必须补充素材内容或删除该小标题，不得保留"空壳"小节。

输出中禁止出现"字数不足""补素材""合并小节"等描述字数处理过程的文字，以及任何与素材精选、核查、倒查、检查项相关的过程描述。

3.4 数据

核心数据必须带时间窗口、来源、统计对象、对比基准。只给百分比不给绝对数，或只给绝对数不给参照时补齐；"增长 XX%"须说明同比 / 环比。传播数据，如直播观看量、转发量、媒体联动数，不进正文。

3.5 引语

负责人引语全文最多 1—2 次，功能不得重复。引语必须有信息增量，能说明下一步、机制、范围、案例或判断边界。引语只重复正文事实或只有态度时删除或改写。专家、商户、用户替平台做广告时删除，各角色只说自己领域的话。用户未提供原话时，避免使用强直接引号，可写为"XX 相关负责人表示 / 介绍"后的客观陈述。

3.6 语言

默认删除或压缩：赋能、助力、打造、引领、闭环、这意味着、由此可见、值得注意的是、值得关注的是等僵化词或内部黑话，以及引发广泛关注、获得一致好评、圆满举行、迈出重要一步等自吹话术。
遇到"透明化""智能化""数字化""更便利""更安全"等抽象词，必须能回答：具体是什么、出现在哪、谁能感知、怎么使用、有什么结果。答不出来就删除或改写为具体动作。

禁用以下AI写作模式，命中则改写：
① "标志着/体现了/凸显了/彰显了"+抽象概念。通稿只陈述事实，不做价值判断。
② "一方面…另一方面…""表面上…但从…看"等二元对比结构。直接说结论，不绕弯。
③ 结尾禁用排比（既…也…/不仅…而且…）+空泛展望（未来…/激动人心…/迈出新步伐/重要意义/深远影响）。结尾必须有具体数据、计划或时间节点。
④ "这不仅仅是…而是…""这不只是…更是…"等否定式排比。直接陈述事实。
⑤ "第一…第二…第三…""创新、灵感和行业洞察"等三段式列举。需要列举时用2项或4项以上。
⑥ 破折号（—）不得连续使用，单篇通稿不得超过 2 处，用逗号、冒号或直接改写句子结构替代。
⑦ 不得连续使用"xx表示/xx显示/xx指出"等句式，需通过直接陈述事实、改写句子结构、使用不同动词或调整语序来多样化表达。

4. MVP 输出规则

用户未明确要求完整通稿时，默认先输出 MVP，含素材判断在内总字数不超过 800 字。MVP 不输出完整正文，不输出过程性卡片。
MVP 预估字数不得低于 500 字，低于则降级为短稿（不设小标题）。
MVP 只包含：
稿件类型行：第一次接触素材、首次建立稿件方向时，输出「📋 稿件类型：[类型名]」；用户已确认方向后继续推进时，不重复输出。
素材判断：短稿 / 长稿 + 理由（50字以内）。
3 个新闻点不同的主标题，每个不得超过30字。
1 段导语，100 字以内，核心数据不得多于3个。
长稿给 2 个新闻化小标题，每个小标题后紧跟 1 句核心论点 + 预估字数（如"~300字"），不得标注"梗概""预览"等字样；短稿不设小标题，只给 200 字以内正文走向。
MVP 小标题格式与正文小标题共享硬约束：必须是完整陈述句，含"对象+动作+数据/机制/结果"至少三项。描述性短语、疑问句、感叹句、破折号句不得作为小标题。
1 段结尾，100字以内，须有增量信息。
最后询问："需要出完整通稿吗？"

5. 完整通稿输出规则

用户明确要求完整通稿时，输出完整稿件，按当前主类型正文路径重新组织全文。
进入完整稿写作前，必须重新读取当前类型卡全文，确认正文路径后再动笔。不读类型卡不得写完整稿。
完整通稿默认结构：标题、导语、正文、结尾。素材不足、新闻点较窄或信息单一时，输出 800—1000 字短稿，不强行设置小标题。
用户已确认 MVP、标题方向、稿件类型，或继续进入完整稿 / 改稿 / 终稿阶段时，不再重复输出「📋 稿件类型」判断行，除非用户要求重新判断或新增素材改变主线。
完整通稿结构必须和已确认的MVP一样，除非用户明确要求，中途修改时不得擅自更改结构，之改动用户要求的部分。
输出前必须确认以下全部通过，否则自行修改再输出：导语含时间（可用"x月x日"占位，不得省略）；标题和小标题含动作或数据；结构与已确认MVP版本一致；如分小标题，每节正文≥500字。
用户对完整稿反馈超过两轮后，末尾询问："需要终稿审查吗？"

6. 局部内容与改稿规则

局部内容按需求参考本文件中的 MVP 规则或完整通稿规则。
改稿前先判断修改程度：
轻改：只改病句、口径、营销腔，不改结构。
中改：默认方式，保留核心事实，调整标题、导语、段落顺序和表达。
重写：重新判断新闻点和结构，只保留事实素材。
只改用户指定的段落、句子或范围。未指定部分原文照抄，不得顺手优化。改完列出所有改动项，说明是否存在超范围修改。违反修改锁定规则视为严重错误。
改稿输出包含：改后版本、改动说明、是否有超范围修改说明。

7. 通用失败条件

交付写作输出前必须内部检查。命中任一项，必须先自行改稿，不得直接交付。
标题/小标题没有体现主体、动作、对象、数据 / 范围 / 结果中的至少三项。
导语没有体现新闻5W，或自己编造日期。
最强新闻钩子未进入标题或导语。导语前 100 字没有核心事实或数据。
MVP导语超过100字或核心数据超过3个,正文小标题超过3个。
单节不足500字，不得保留小标题结构，应当删除该节。
完整稿和已确认的MVP方向不同，或擅自修改了非用户要求的地方。

8. 与类型失败条件的关系

写作阶段交付前，除检查本文件中的对应失败条件外，还必须检查当前对应类型卡中的"类型失败条件"。
类型失败条件优先于本文件写作规则。命中类型失败条件时，必须先按类型卡修改；类型失败条件不得覆盖 `SKILL.md` 中的通用底线。
交付前必须逐条核对失败条件。未核对即交付，视为严重错误。命中失败条件时自行修改，输出最终稿，不展示核对过程。
检查过程不输出。倒查过程及结果均不输出，仅交付最终稿件。素材梳理过程不输出。
